# Game Tests (Template Monorepo)

A game built from `nexus new mygame` inherits the engine's complete test stack. Same tools. Same commands. Same CI. Same gates.

Game devs write **gameplay scenarios** and **gameplay unit tests**. They never configure a test runner.

→ `docs/game-template/overview.md`, `docs/game-template/structure.md`.

## What the template ships

```
mygame/
├── game/
│   ├── src/
│   ├── tests/

│   │   └── inventory.rs
│   └── benches/
├── server/
│   ├── src/
│   └── tests/
├── web/
│   ├── src/
│   └── tests/
├── mobile/
│   └── tests/
├── ai-agents/
│   ├── src/
│   └── tests/
├── dlc/
├── mods/
├── scenarios/                       # gameplay scenarios (TOML)
│   ├── core-loop/
│   │   ├── tutorial-completes.toml
│   │   └── first-boss-defeated.toml
│   └── regression/
│       └── door-glitch-fixed.toml
├── replays/                         # curated regression replays (LFS)
├── visual/                          # golden frames (LFS)
├── nexus.toml

└── .github/workflows/
    └── nexus-ci.yml                 # identical to engine CI, scoped to game
```

Every config file is generated by `nexus new`. Identical content to the engine. No edits needed.

## One command per concern

```bash
nexus test                # all unit + integration (Rust + TS + Python + Lua)
nexus test --scenario     # all TOML scenarios
nexus test --visual       # visual diffs
nexus test --network      # multi-client / lag-injection
nexus test --replay       # curated regression replay suite
nexus test --coverage     # coverage report (per crate floor enforced)
nexus bench               # criterion + vitest bench + pytest-benchmark
nexus fmt                 # all formatters
nexus lint                # all linters
nexus fuzz                # smoke fuzz on touched targets
nexus ci                  # the full pipeline locally
```

Behind the scenes: `nexus test` orchestrates `cargo nextest`, `vitest`, `pytest`, `busted`, and the scenario runner.

## What a game dev writes

| Test class | Where | When |
|------------|-------|------|
| Gameplay scenario | `scenarios/<feature>/*.toml` | every new gameplay feature |
| Unit test (Rust gameplay) | `game/src/**` `#[cfg(test)]` | every public game-side function |
| Integration (server) | `server/tests/*.rs` | every server endpoint |
| Lua script test (busted) | `game/scripts/*_spec.lua` | every Lua module |
| Visual regression | scenario with `[capture]` | every renderer-sensitive feature |
| Replay (regression) | `replays/<bug-id>.replay` | every shipped bug fix |
| Property | `tests/prop_*.rs` | math/serialization in game code |
| Bench | `game/benches/*.rs` | perf-sensitive systems |
| Visual | `visual/<scenario>/.../*.png` | new UI / shader |
| Fuzz | `game/fuzz/` | any save-file or mod-loader format |

Game-side **does NOT write**:
- engine fixtures (reuse `nexus_test::*`)
- CI workflow (template ships it)
- linter configs (template ships them)
- coverage runners (template wires them)
- visual diff tooling (engine ships it)

## Example: gameplay scenario

```toml
# scenarios/core-loop/tutorial-completes.toml
schema = "nexus.scenario/v1"
name   = "tutorial completes end-to-end"
seed   = 1
tags   = ["smoke", "core-loop"]

[engine]
backend  = "noop"
audio    = "null"
clock    = "frozen"
plugins  = ["nexus-genre-rpg"]
scene    = "scenes/tutorial.scene"

[steps]
1.spawn       = { archetype = "Player", at = [0, 0, 0] }
2.run_frames  = 60
3.apply_input = { player = 1, input = "interact" }                # talk to NPC
4.run_frames  = 30
5.apply_input = { player = 1, input = "move_forward", frames = 120 }
6.run_frames  = 120
7.apply_input = { player = 1, input = "interact" }                # pick up sword
8.run_frames  = 30

[assert]
"world.entities.Player[0].inventory.contains[*].kind" = { contains = "sword" }
"world.quests.tutorial.status"                        = "completed"
"telemetry.errors.length"                              = 0
```

`nexus test --scenario scenarios/core-loop/` runs this in < 5 s. Asserts on the actual gameplay state — survives any refactor of how the tutorial is wired internally.

## Example: gameplay unit test

```rust
// game/src/combat/damage.rs
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn crit_doubles_damage() {
        let attack = Attack { base: 10, crit_chance: 1.0, crit_mult: 2.0 };
        let target = Target::default();
        let dmg = compute_damage(&attack, &target, &mut rng_for_test("crit_doubles"));
        assert_eq!(dmg, 20);
    }

    #[test]
    fn armor_reduces_damage_by_percent() {
        let attack = Attack { base: 100, crit_chance: 0.0, crit_mult: 1.0 };
        let target = Target { armor_pct: 0.25, ..Target::default() };
        let dmg = compute_damage(&attack, &target, &mut rng_for_test("armor"));
        assert_eq!(dmg, 75);
    }
}
```

## Example: visual scenario

```toml
# scenarios/visual/main-menu.toml
schema = "nexus.scenario/v1"
name   = "main menu renders golden"
seed   = 0

[engine]
backend  = "vulkan"
clock    = "frozen"
scene    = "scenes/main-menu.scene"

[steps]
1.run_frames = 1

[capture]
frame      = 1
output     = "main-menu.png"
resolution = [1920, 1080]
tolerance  = { strategy = "exact" }

[assert]
"capture.frame.matches_golden" = true
```

Golden lives at `visual/main-menu/vulkan/1920x1080.png` (LFS). → `visual.md`.

## CI inheritance

Game template ships `.github/workflows/nexus-ci.yml` — byte-identical to `crates/nexus-test/templates/nexus-ci.yml` in the engine repo, with the `<GAME_NAME>` placeholder substituted by `nexus new`.

It runs:

```
fmt → lint → unit → integration → scenario → visual → network → perf → coverage
```

Same gates. Same JSON output. Same nexus-merge consumption.

→ `ci.md`.

## Custom assertions via game DSL

Games can register custom assertion verbs:

```rust
// game/src/test_dsl.rs
nexus_test::register_assertion!("game.player.has_item", |args, world| {
    let player = world.singleton::<Player>();
    let item   = args.required_str("item")?;
    Ok(player.inventory.contains(item))
});
```

Now scenarios can write:

```toml
[assert]
"game.player.has_item" = { item = "sword" }
```

## Mods and DLC

Mods (`mods/*`) and DLC (`dlc/*`) are crates. They get the same:
- tests folder
- scenarios folder (`mods/<name>/scenarios/`)
- CI matrix entry
- coverage floor

Mod scenarios run against a base game + mod loaded. Engine guarantees sandboxing → `docs/specs/scripting/sandbox.md`.

## AI-agent self-tests

`ai-agents/` ships with tests by default:

| Test | What |
|------|------|
| Agent boots, connects to headless engine | smoke |
| Agent receives scenario telemetry | telemetry contract |
| Agent successfully spawns + queries entity | API contract |
| Agent's playtest scenarios pass | the agent IS a test runner |

→ `docs/specs/agent/sdk.md`.

## Per-game coverage floor

Defaults inherited from engine, overridable in `nexus.toml`:

```toml
[test.coverage]
default        = 70
per_crate."game"     = 75
per_crate."server"   = 80
per_crate."ai-agents"= 50
per_crate."mods/*"   = 60
```

→ `coverage.md`.

## Hard rules

| Rule | |
|------|--|
| Every gameplay feature ships with a scenario | template enforces via PR checklist |
| Every shipped bug fix ships with a regression replay or scenario | block-on-merge |
| Game devs never edit linter/test configs | nexus-merge rejects config drift |
| Game CI is identical to engine CI (template-substituted) | drift → engine PR |
| Custom assertion verbs registered, not hand-rolled | greppable surface |
| Save-file format has fuzz target | untrusted input |
| Mod sandbox boundary has fuzz target | untrusted input |
| Replay schema matches engine version | save game = replay = scenario |

## Forbidden

| Pattern | Why |
|---------|-----|
| Hand-edited CI workflow | drift from template |
| Custom test runner | use `nexus test` |
| Wall-clock scenario duration > 60 s | move to perf bucket |
| Visual goldens outside LFS | repo bloat |
| Game-local fixtures replicating `nexus_test::*` | DRY |
| Tests that read player's local FS | hermetic |
| `#[ignore]`'d tests without ticket | dead test |
